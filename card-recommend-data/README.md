# card-recommend-data

카드 추천·결제 앱의 추천 알고리즘 백엔드용 카드 데이터베이스.
카드고릴라 크롤링 원본(1,558개 카드)을 전처리하여 정형화한 데이터와
추천 스코어링 레퍼런스 구현을 포함한다.

- 데이터 스냅샷: **2026-07-14**
- 카드 1,558건 / 혜택 6,981건 / 가맹점 사전 시드 624건

## 저장소 구조

```
data/
  card_database.json        # 메인 DB — 카드 안에 혜택 배열이 중첩된 문서형 구조
  cards_master_final.csv    # 카드 마스터 테이블 (RDB 적재용)
  card_benefits_final.csv   # 혜택 상세 테이블 + 혜택 원문 전문 (JSON에는 원문 미포함)
  merchant_dictionary_seed.csv  # 가맹점명 → 주카테고리/혜택보유카드수 (결제 시점 매칭 시드)
docs/
  data_spec.md              # 컬럼 정의서 (타입·의미·결측 규칙)
src/
  recommend_prototype.py    # 추천 스코어링 레퍼런스 구현 (API 포팅 명세)
```

## 테이블 관계

- `cards_master` 1 : N `card_benefits`, 조인 키 = `카드번호`
- 혜택 테이블 PK = `혜택ID` (형식: `{카드번호}-{순번2자리}`)
- JSON은 위 두 테이블을 카드 단위로 중첩한 구조이며, 용량 관리를 위해
  혜택 `상세`(원문 전문)는 CSV에만 존재한다. 상세화면 조회는 `혜택ID`로 CSV 측 참조.

## 스코어링 핵심 규칙 (src/recommend_prototype.py 참조)

1. **월 기대혜택** = Σ min(혜택율 × 해당 카테고리 소비액, 월최대혜택액) − 연회비_최소/12
2. **실적 조건**: 카드 공통 `전월실적` + 혜택별 `실적조건` 모두 충족해야 해당 혜택 적용
3. **옵션그룹(택1)**: 같은 `옵션그룹` 값의 혜택은 **합산 금지** — 그룹 내 기대혜택 최대 1개만 반영.
   `옵션헤더=True` 블록과 `적용범위='연1회기프트'` 블록은 월 단위 스코어링에서 제외
4. **다중 카테고리**: `카테고리목록`이 있으면 해당 카테고리들의 소비액 합산에 적용.
   카테고리 `모든가맹점`은 총 소비액에 적용
5. **가맹점 커버리지**: 월 기대혜택 계산 시 `가맹점수준`별 계수를 곱한다
   (업종전체 1.0 / 복수가맹점 0.8 / 단일가맹점 0.4 / 불명 0.5 — 파라미터).
   단, **결제 시점 추천**에서는 계수 대신 `가맹점목록` 직접 매칭을 사용할 것
6. **한도 미상 처리**: `월최대혜택액` NULL이고 `한도없음=False`인 혜택은
   카드 `통합한도_월`을 캡으로 쓰고, 그것도 없으면 할인율 20% 이상에 한해
   보수적 캡 10,000원/월 적용 (과대평가 방지)

## 계산 가정 (JSON meta에도 동일 기재)

| 항목 | 값 |
|---|---|
| 포인트 가치 | 1P = 1원 |
| 마일리지 가치 | 1마일 = 15원 |
| 주유 원/L 혜택 환산 유가 | 1,700원/L |
| 한도 미상 고할인 보수 캡 | 10,000원/월 |

## 알려진 한계 (백엔드 정책 결정 필요)

- 할인율 20% 이상 혜택의 약 45%는 **원문에 한도 수치가 없음**(카드사 공식 페이지에만 존재).
  보수적 캡으로 통제 중이며, 2차 소스(카드사 공식/여신금융협회) 결합이 후속 과제.
  카드 단위 신뢰도는 `한도파악률`(0~1), 혜택 단위는 `한도확보`(bool)로 제공 —
  낮은 카드의 노출/감점 정책은 기획 결정 사항
- 단종/판매중단 여부 컬럼 없음 (재크롤링 필요) — 스키마에 nullable로 예약 권장
- 연회비 면제 조건, MCC 업종코드, 혜택 유효기간 미보유 — 동일하게 예약 권장
- 카드 혜택은 수시 개편되므로 `스냅샷일자` 기준 버전 관리 필수.
  데이터 갱신은 `data/snapshot-YYYY-MM` 브랜치 → PR로 진행 (스키마 변경 시 PR에 명시)
