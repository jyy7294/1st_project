# PICKA 백엔드 중간 발표 정리

## 1. 백엔드 기술 스택

- 언어: Python
- API 서버: FastAPI
- 데이터베이스: Supabase PostgreSQL
- ORM: SQLAlchemy
- DB 마이그레이션: Alembic
- DB 드라이버: Psycopg
- 데이터 검증: Pydantic
- 인증: JWT 일반 로그인
- LLM: OpenAI Responses API
- API 문서: Swagger (`/docs`)
- 테스트: Python unittest, 현재 102개 통과

> FastAPI로 REST API를 만들었고 데이터는 Supabase PostgreSQL에 저장했습니다. SQLAlchemy로 DB를 조회하고 Alembic으로 테이블 변경 이력을 관리했습니다.

## 2. 전체 백엔드 구조

```text
프론트 요청
    ↓
FastAPI
    ↓
사용자·카드·거래 데이터 조회
    ↓
가맹점 및 카테고리 정규화
    ↓
사용자 자격 필터
    ↓
카드 혜택·실적·한도 계산
    ↓
카드 순위 결정
    ↓
추천 결과 반환 및 감사 로그 저장
```

## 3. 주요 DB 구조

### 사용자 관련

- `users`: 기본 사용자 계정
- `user_persona_profiles`: 성별, 직업, 거주지 등 페르소나 정보
- `user_eligibilities`: 병역, 사업자, 외국인, 차량, 자녀, 통신사 등 자격정보
- `user_cards`: 사용자가 보유한 카드

### 카드 관련

- `cards`: 카드 기본정보, 연회비, 전월 실적, 월 통합한도
- `card_benefits`: 카드별 혜택
- `benefit_tiers`: 실적 구간별 혜택
- `card_eligibility_rules`: 카드 발급·추천 필수 자격
- `card_benefit_eligibility_rules`: 멤버십 등 혜택별 자격조건

### 결제·집계 관련

- `transactions`: 실제 카드 결제 이력
- `transaction_rewards`: 포인트·마일리지 적립
- `transaction_benefit_outcomes`: 실제 혜택, 예상 혜택, 놓친 혜택
- `monthly_card_usage`: 카드별 월 사용액과 혜택 집계
- `benefit_usage`: 개별 혜택의 월 사용량
- `merchant_aliases`: 가맹점명과 카테고리 표준화

### 추천 관련

- `card_recommendation_snapshots`: 신규카드 추천 캐시
- `recommendation_audit_logs`: 추천 당시 입력값과 계산 결과 이력

## 4. 보유카드 결제 추천

사용자가 결제처와 금액을 입력하면 보유카드별로 다음을 계산합니다.

```text
가맹점 확인
    ↓
카드별 적용 가능한 혜택 확인
    ↓
전월 실적 충족 여부 확인
    ↓
개별 혜택 한도 적용
    ↓
카드 월 통합한도 적용
    ↓
예상 혜택 계산
    ↓
혜택과 실적을 기준으로 추천
```

### 실적 개념

- 전월 실적: 이번 달에 혜택을 받을 수 있는지 결정
- 이번 달 실적: 다음 달 혜택 자격을 위해 누적
- 이번 결제 금액: 이번 달 실적에 추가
- 실제 할인액: 개별 혜택 한도와 카드 월 통합한도를 모두 적용

월 통합한도는 다음과 같이 처리합니다.

```python
applied_benefit = min(
    calculated_benefit,
    payment_amount,
    monthly_total_limit - current_month_benefit_used,
)
```

월 혜택 한도가 30,000원이고 이미 28,200원을 사용했다면, 계산상 1,900원 할인이더라도 남은 한도인 1,800원만 적용합니다.

## 5. 신규카드 추천

신규카드는 최근 90일 소비패턴을 분석합니다.

```text
최근 90일 승인 거래 조회
    ↓
최근 30일 50%, 이전 30일 30%, 이전 30일 20% 가중치
    ↓
자주 결제한 소비 분야 우선
    ↓
빈도가 비슷하면 사용 금액으로 비교
    ↓
사용자 자격에 맞지 않는 카드 제외
    ↓
남은 카드의 예상 혜택 계산
    ↓
연회비를 반영해 최종 순위 결정
```

### 자격 필터 예시

- 병역 대상이 아님 → 나라사랑카드 제외
- 내국인 → 외국인 전용카드 제외
- 자녀 1명 → 다자녀카드 제외
- 학생이 아님 → 학생증카드 제외
- 경차 미보유 → 경차 유류세 환급카드 제외

현재 전체 카드 1,559장을 저장했고, 병역·외국인·연령·학생·사업자·복지·차량·K-패스 등의 특수 조건을 별도 규칙으로 관리합니다.

## 6. LLM 사용 범위

LLM은 추천 금액이나 카드 순위를 직접 결정하지 않습니다. 정형화된 규칙으로 판단하기 어려운 카드 약관만 보조 판단합니다.

예시는 다음과 같습니다.

- 일부 입점 매장 제외
- 특정 제휴 가맹점만 제공
- 일반 상품은 가능하지만 상품권은 제외

### LLM 입력

- 가맹점명
- 결제 카테고리
- 결제 금액
- 혜택명
- 약관 문구
- 구조화된 혜택 조건

### LLM 출력

OpenAI Responses API의 JSON Schema를 사용해 출력 형식을 제한합니다.

```json
{
  "applicable": true,
  "confidence": 0.85,
  "reason": "해당 가맹점이 혜택 대상에 포함됩니다.",
  "needs_human_review": false,
  "caveat": ""
}
```

> 할인액 계산, 실적 계산, 한도 적용, 최종 순위는 모두 백엔드 규칙으로 처리합니다. LLM은 모호한 문구의 적용 가능성만 판단하기 때문에 결과의 재현성과 안정성을 확보했습니다.

LLM 호출에 실패해도 추천 서버 전체가 실패하지 않습니다. 해당 혜택을 조건부로 표시하고 사용자 확인이 필요하다고 반환합니다. 동일한 입력은 메모리 캐시로 재사용합니다.

## 7. 데이터 정합성 처리

### 가맹점 분류 통일

`merchant_aliases`를 단일 기준으로 사용합니다.

```text
교보문고
교보문고 강남점
교보문고 온라인몰
교보문고 문구
→ 모두 영화/문화
```

프론트에서 다른 카테고리를 보내더라도 등록된 alias가 우선합니다.

### 혜택 합계 통일

모든 화면에서 승인 완료 거래의 `transactions.saved_amount` 합계를 혜택 합계 기준으로 사용합니다.

합계에서 제외하는 값:

- 추천 단계의 예상 혜택
- 놓친 혜택
- 취소 거래
- 원화와 단위가 다른 포인트·마일리지

### 원본과 시연 결제 구분

```text
transactions.data_source
- SEED: 원본 페르소나 거래
- DEMO: 발표 중 실행한 시연 결제
```

`demo_session_id`로 시연 회차도 구분합니다. 월 합계 화면에는 `SEED + DEMO` 카드 거래를 함께 표시하고 비카드 자동이체는 저장하지 않습니다.

## 8. 추천 캐시와 감사 로그

### 추천 캐시

신규카드 추천은 계산량이 많아 하루 단위로 캐시합니다.

- 사용자별·분석일별 저장
- 추천 정책 버전이 바뀌면 자동 재계산
- 사용자 자격이 수정되면 캐시 삭제
- 캐시 결과도 반환 직전에 현재 자격으로 다시 검사

### 감사 로그

추천 요청마다 다음 정보를 저장합니다.

- 요청 사용자
- 결제처와 금액
- 카드별 계산 결과
- 제외된 카드와 이유
- 최종 추천 카드
- 적용한 추천 정책 버전
- 캐시 사용 여부

문제가 발생하면 당시 계산을 재현할 수 있으며, 로그는 90일이 지나면 자동 삭제합니다.

## 9. 주요 API

```text
GET  /api/v1/users/{id}/cards

POST /api/v1/recommendations
결제 전 최적 보유카드 추천

POST /api/v1/transactions
시연 결제 및 혜택 저장

GET  /api/v1/users/{id}/card-recommendations
최근 소비패턴 기반 신규카드 추천

GET/PUT /api/v1/users/{id}/eligibilities

```

Swagger 실행 주소:

```text
http://localhost:8000/docs
```

## 10. 발표용 요약 대본

> PICKA 백엔드는 FastAPI와 Supabase PostgreSQL로 구현했습니다. 사용자의 보유카드, 결제 이력, 카드 혜택, 전월 실적과 월 한도를 DB에서 관리합니다.
>
> 결제 전 추천은 보유카드별 예상 혜택과 실적을 계산해 최적 카드를 제안합니다. 신규카드 추천은 최근 90일 소비 빈도와 금액을 분석한 뒤 사용자 자격에 맞지 않는 특수카드를 먼저 제외합니다.
>
> 추천 금액과 순위는 규칙 기반으로 계산하며, LLM은 카드 약관의 모호한 문구를 판단하는 보조 역할만 수행합니다. 또한 추천 캐시, 계산 감사 로그, 가맹점 카테고리 표준화와 월 통합한도 검증을 적용해 결과의 일관성과 추적 가능성을 확보했습니다.

## 11. 예상 질문과 답변

### Q. 왜 LLM이 직접 추천하지 않나요?

금액과 순위를 LLM에 맡기면 같은 입력에서도 결과가 달라질 수 있기 때문에 핵심 계산은 규칙 기반으로 구현했습니다.

### Q. LLM은 꼭 필요한가요?

명확한 혜택은 LLM 없이 계산합니다. 가맹점 포함 여부처럼 약관만으로 판별하기 어려운 경우에만 사용합니다.

### Q. 특수카드는 어떻게 제외하나요?

사용자 자격과 카드 필수 자격을 DB에 각각 저장하고, 신규카드 후보 단계에서 비교해 미충족 카드를 제외합니다.

### Q. 추천이 잘못되면 어떻게 확인하나요?

추천 당시 입력, 후보별 계산값, 제외 이유와 최종 순위를 감사 로그에 저장해 원인을 추적합니다.

### Q. 백엔드에서 가장 신경 쓴 부분은 무엇인가요?

전월 실적과 이번 달 실적을 구분하고, 개별 혜택 한도와 카드 월 통합한도를 실제 결제 저장 시점까지 강제한 부분입니다.

### Q. LLM 호출이 실패하면 추천도 실패하나요?

아닙니다. 해당 혜택을 조건부로 처리하고 사용자 확인이 필요하다고 표시하며, 규칙 기반 추천 과정은 계속 진행됩니다.
